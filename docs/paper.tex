% ─── Abstract ─────────────────────────────────────────────────────────────────
\begin{abstract}
\noindent
We present \textbf{Israel Shield} (\texthebrew{מגן ישראל}), a browser-based, real-time security intelligence dashboard that aggregates, analyzes, and visualizes over seven years of rocket, missile, drone, and terrorist-infiltration alerts across Israel. The system integrates a live feed from the IDF Home Front Command with a curated historical dataset of 10{,}000+ records spanning 2019--2026, processing them through a suite of statistical and algorithmic models. Key contributions include: (1) a \emph{Spatio-Temporal UAV Route Reconstruction} algorithm that chains drone alert sequences into probable flight trajectories; (2) a \emph{Safe Route Departure Advisor} that scores every departure hour using a sliding-window aggregation over historical threat corridors along a travel path; (3) a \emph{Shower Index} (\texthebrew{מדד המקלחת}), a Poisson-based daily safety-window estimator; and (4) a fully client-side, offline-capable Progressive Web App supporting seven languages and bidirectional text. The system demonstrates that meaningful security analytics are achievable without proprietary infrastructure, enabling both civilian and academic use cases.

\smallskip
\noindent\textbf{Keywords:} security analytics, threat visualization, spatio-temporal analysis, UAV tracking, safe route planning, business intelligence, homeland security, Israel.
\end{abstract}

\section{Introduction}
\label{sec:introduction}
% ═════════════════════════════════════════════════════════════════════════════

Since 2001, Israel has faced persistent rocket fire from Gaza and Lebanon. From October 2023, an unprecedented multi-front escalation added ballistic missiles, cruise missiles, and large-scale UAV swarms from Iran, Yemen, Iraq, and Syria simultaneously. The IDF Home Front Command (\textit{Pikud HaOref}, \texthebrew{פיקוד העורף}) operates a public alert system that has generated a historically unique dataset: a timestamped, city-level record of every incoming-threat warning issued to the Israeli civilian population.

This dataset presents a rich opportunity for \emph{threat pattern analysis} — the systematic study of \emph{when}, \emph{where}, and \emph{how} security incidents occur at population scale. Despite its public availability, no accessible multi-functional analytical tool existed combining live feeds, historical analysis, geographic visualization, probabilistic safety estimation, and navigation planning in a single interface.

\textbf{Israel Shield} fills this gap. It is a full-stack web application built as a Business Intelligence project at Ariel University. Its key distinguishing features are:

\begin{enumerate}
  \item \textbf{UAV Route Reconstruction} — a novel spatio-temporal chaining algorithm reconstructing probable drone flight trajectories from ordered city-alert sequences.
  \item \textbf{Safe Route Planner} — a departure time optimizer scoring each departure hour by historical threat intensity along a proposed travel corridor.
  \item \textbf{Shower Index} — a Poisson-based daily safety window estimator.
  \item \textbf{Seven temporal resolutions} of alert frequency analysis (year → 1-minute resolution).
  \item \textbf{Full offline capability} via IndexedDB caching and Progressive Web App (PWA) support.
  \item \textbf{Seven languages} with bidirectional text (Hebrew/Arabic RTL).
\end{enumerate}

\section{Background and Related Work}
\label{sec:background}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{Open Security Data in Israel}
The Home Front Command's ``Red Alert'' (\texthebrew{צבע אדום}) system is one of the world's most densely documented civilian warning datasets, timestamped to the second at city level since 2001. Academic use has been limited; most analyses are journalistic \cite{harpaz2023}.

\subsection{Threat Visualization Platforms}
Existing tools (LiveUA Map, ACLED) map macro-level conflict without city-level temporal granularity. None integrate the IDF alert feed's precision or provide civilian-facing safety planning.

\subsection{UAV Trajectory Reconstruction}
Trajectory inference from indirect observations is active in air traffic management \cite{olive2019}. Prior work requires ADS-B or radar data. This paper presents the first method using civilian warning activations as trajectory proxies.

\subsection{Route Safety Planning}
Route safety in conflict zones has been studied in humanitarian logistics \cite{tomasini2009}. Most approaches optimize \emph{which road} to take; our approach optimizes \emph{when} to travel on any road.

\subsection{Poisson Models for Safety Windows}
Poisson processes are standard models for rare event occurrence in time \cite{kingman1992}, applied here to compute zero-alert interval probabilities.

\section{Data Sources and Processing Pipeline}
\label{sec:data}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{Data Sources}

\subsubsection{Historical Alert Dataset}
The primary source is a community-curated CSV by Dr.\ Yuval Harpaz \cite{harpaz2023}. Each record contains:

\begin{table}[H]\centering
\caption{Alert Record Schema}
\begin{tabular}{lll}\toprule
\textbf{Field} & \textbf{Type} & \textbf{Description} \\\midrule
\texttt{time}     & String & ISO timestamp (\texttt{2023-10-07 06:29:00}) \\
\texttt{cities}   & String & Hebrew city/region name \\
\texttt{threat}   & String & Numeric code or Hebrew category \\
\texttt{category} & String & Alternative threat field \\
\texttt{id}       & String & Optional IDF event batch ID \\\bottomrule
\end{tabular}
\end{table}

\noindent The dataset contains $N\!\approx\!10{,}000{+}$ records spanning 2019--2026, with density increasing dramatically from October~7, 2023.

\subsubsection{Live Alert Feed}
Real-time alerts are fetched from \texttt{oref.org.il} via an \texttt{allorigins.win} CORS proxy at 10-second intervals.

\subsubsection{Geographic Coordinates}
A \texttt{baseCoords} dictionary hardcodes coordinates for 450+ Israeli cities, settlements, and regions. For unknown cities, the OpenStreetMap Nominatim API is queried with rate-limiting (1 req/s, 20 cities/batch), and results cached in \texttt{localStorage}.

\section{UAV Route Reconstruction Algorithm}
\label{sec:uav}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{Motivation}
When a hostile UAV (\texthebrew{כטב"מ}) enters Israeli airspace, Home Front Command activates alerts in cities sequentially as it approaches each population center. The time-ordered city alert sequence implicitly encodes the flight trajectory.

\subsection{Problem Formulation}

\begin{definition}[UAV Alert Sequence]
Let $\mathcal{A} = \{a_1,\ldots,a_n\}$ be all ``hostile aircraft intrusion'' alerts, sorted chronologically. Each $a_i=(t_i,\mathbf{p}_i)$ where $t_i$ is the timestamp and $\mathbf{p}_i=(\phi_i,\lambda_i)$ are geographic coordinates.
\end{definition}

\begin{definition}[UAV Route]
A route $R=(a_{i_1},\ldots,a_{i_k})$ is a maximal subsequence satisfying for each consecutive pair:
\[
t_{i_{j+1}}-t_{i_j}\leq\tau_{\max}=10\text{ min}
\quad\text{and}\quad
d_H(\mathbf{p}_{i_j},\mathbf{p}_{i_{j+1}})\leq d_{\max}=20\text{ km}
\]
\end{definition}

\subsection{Coordinate Smoothing}

A 1-2-1 weighted convolution removes zigzag artifacts from simultaneous multi-city alerts:
\begin{equation}
\hat{\mathbf{p}}_i = \frac{\mathbf{p}_{i-1}+2\mathbf{p}_i+\mathbf{p}_{i+1}}{4}
\end{equation}
with endpoints fixed ($\hat{\mathbf{p}}_0=\mathbf{p}_0$, $\hat{\mathbf{p}}_k=\mathbf{p}_k$). Two passes are applied.

\subsection{Segment Frequency Scoring}

For each city pair $(c_i,c_j)$ appearing across all routes, frequency $f_{ij}$ drives visual weight:
\[
\text{opacity}_{ij}=0.3+0.7\cdot\tfrac{f_{ij}}{f_{\max}}, \qquad
\text{weight}_{ij}=2+4\cdot\tfrac{f_{ij}}{f_{\max}}
\]

\section{Safe Route Departure Planner}
\label{sec:route}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{Problem Statement}
Given origin $O$ and destination $D$, identify the departure hours $h^*\in\{0,\ldots,23\}$ that minimise expected historical alert exposure during travel.

\subsection{Algorithm}

\begin{figure}[H]\centering
\begin{tikzpicture}[
  step/.style={rectangle, rounded corners=4pt, draw=israelblue!70, fill=israelblue!8,
               minimum width=8.8cm, minimum height=0.85cm, align=center, font=\small},
  arr/.style={-Stealth, thick, israelblue!60},
  node distance=0.85cm,
]
  \node[step] (s1) {Geocode $O$, $D$ from \texttt{baseCoords} / Nominatim};
  \node[step, below=of s1] (s2) {OSRM API: road distance $d_r$ and duration $\Delta T$\\(6\,s timeout; fallback: $d_r\!=\!1.35\,d_H$, $\Delta T\!=\!d_r/80$)};
  \node[step, below=of s2] (s3) {Interpolate straight-line path every 5\,km ($\sim$30--150 points)};
  \node[step, below=of s3] (s4) {Find impact zone: all cities within 15\,km of any path point};
  \node[step, below=of s4] (s5) {Aggregate per-hour alert counts $c_h$ for impact zone cities};
  \node[step, below=of s5] (s6) {Score each of 24 departure hours: $S(h)=\texttt{getWindowScore}(c,h,\Delta T)$};
  \node[step, below=of s6] (s7) {Rank all 24 hours; output top 12 with arrival time \& quiet\%};

  \draw[arr] (s1)--(s2)--(s3)--(s4)--(s5)--(s6)--(s7);
\end{tikzpicture}
\caption{Safe Route Departure Planner algorithm.}
\label{fig:route_algo}
\end{figure}

\subsection{Key Equations}

\paragraph{Path interpolation.}
\[
\mathbf{p}(t)=(1-t)\,\mathbf{p}_O + t\,\mathbf{p}_D,\quad
t\in\Bigl\{\tfrac{i}{\lceil d/5\rceil}\Bigr\}_{i=0}^{\lceil d/5\rceil}
\]

\paragraph{Impact zone.}
\[
\mathcal{Z}=\bigl\{c\in\mathcal{C}\mid\exists\,\mathbf{p}\in\mathrm{path}:d_H(\mathbf{p},\mathbf{p}_c)\leq15\,\text{km}\bigr\}
\]

\paragraph{Window score.}
\begin{equation}
S(h,\Delta T)=\sum_{i=0}^{\lfloor\Delta T\rfloor-1}c_{(h+i)\bmod 24}
+c_{(h+\lfloor\Delta T\rfloor)\bmod 24}\cdot\{\Delta T\}
\label{eq:window}
\end{equation}
where $\{x\}$ is the fractional part of $x$.

\paragraph{Relative quiet probability.}
\[
P_{\mathrm{quiet}}(h)=\Bigl(1-\tfrac{S(h)}{\max_{h'}S(h')}\Bigr)\times100\%
\]

\subsection{OSRM Fallback}
If the OSRM query fails (timeout or server error), the application falls back to:
\[
d_r=1.35\,d_H,\qquad \Delta T=d_r/80\text{ h}
\]
The 1.35 multiplier reflects the typical road-to-straight-line distance ratio for Israeli inter-city routes.

% ═════════════════════════════════════════════════════════════════════════════
\section{Shower Index: Daily Safety Window Estimator}
\label{sec:shower}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{Motivation}
Civilians in prolonged operations need to know \emph{when} is the safest 30-minute window for everyday activities that temporarily prevent them from reaching a shelter (showering, cooking, etc.). The Shower Index answers this question statistically.

\subsection{Algorithm}

\begin{enumerate}
  \item \textbf{Bucket} all alerts into 48 half-hour slots:
  $b(a)=\lfloor(60\cdot\mathrm{hour}(a)+\mathrm{min}(a))/30\rfloor\in\{0,\ldots,47\}$

  \item \textbf{Weight} each slot (penalises target slot more than neighbours):
  $w_i=3c_i+c_{(i-1)\bmod48}+c_{(i+1)\bmod48}$

  \item \textbf{Find optimal slot}, preferring waking hours (07:00--22:00):
  \[
  i^*=\begin{cases}\arg\min_{i\in[14,44]}w_i&\text{if such }i\text{ exists with }c_i<\max_j c_j\\\arg\min_{i\in[0,47]}w_i&\text{otherwise}\end{cases}
  \]

  \item \textbf{Poisson probability} of zero alerts in the optimal window:
  \[
  \lambda=\frac{c_{i^*}}{|\mathcal{D}|},\qquad P(\text{quiet})=e^{-\lambda}
  \]
  where $|\mathcal{D}|$ is the number of distinct days in the filtered dataset.
\end{enumerate}

\begin{proposition}
Under a homogeneous Poisson process with rate $\lambda$ per 30-minute window per day, $P(\text{quiet})=e^{-\lambda}$ is the probability of observing zero alerts in a randomly chosen instance of the recommended slot.
\end{proposition}

\section{Implementation Notes}
\label{sec:implementation}
% ═════════════════════════════════════════════════════════════════════════════

\subsection{React Anti-Pattern: Inner Function Components}

A critical bug was identified during development. When React components are defined \emph{inside} a parent render function, each render creates a new component type identity, causing React to unmount and remount it — resetting all internal state including keyboard focus:

\begin{lstlisting}[style=tscode, language=JavaScript,
  caption={Incorrect: inner function component loses focus on each parent re-render}]
const SafeRouteModal = () => {
  const [city, setCity] = useState('');
  // NEW component type identity on every render -> unmount/remount!
  const InputForm = () => (
    <input value={city} onChange={e => setCity(e.target.value)} />
  );
  return <InputForm />;  // Focus lost after every keystroke
};
\end{lstlisting}

\begin{lstlisting}[style=tscode, language=JavaScript,
  caption={Correct: JSX variable preserves identity; focus retained}]
const SafeRouteModal = () => {
  const [city, setCity] = useState('');
  // JSX value, not a new component type -> stable identity
  const inputFormJsx = (
    <input value={city} onChange={e => setCity(e.target.value)} />
  );
  return <div>{inputFormJsx}</div>;
};
\end{lstlisting}

\subsection{ECharts Sizing in Flex Containers}

Apache ECharts reads \texttt{clientHeight} at initialization. In CSS flexbox with \texttt{flex:1} / \texttt{min-height:0}, layout may not be finalized at that moment. The solution is a \texttt{ResizeObserver}:

\begin{lstlisting}[style=tscode, language=JavaScript,
  caption=ResizeObserver pattern for correct ECharts height in flex containers]
const [chartHeight, setChartHeight] = useState(0);
const chartBoxRef = useRef<HTMLDivElement>(null);

useEffect(() => {
  const el = chartBoxRef.current;
  if (!el) return;
  const ro = new ResizeObserver(entries => {
    const h = entries[0]?.contentRect.height;
    if (h && h > 0) setChartHeight(h);
  });
  ro.observe(el);
  return () => ro.disconnect();
}, [safeRouteData]);

// JSX:
<div ref={chartBoxRef} style={{ flex: 1, minHeight: 0 }}>
  <ReactECharts style={{ height: chartHeight - 16 }} />
</div>
\end{lstlisting}

\section{Conclusion}
\label{sec:conclusion}
% ═════════════════════════════════════════════════════════════════════════════

We have presented Israel Shield, a browser-based security intelligence dashboard that transforms a decade of public alert data into actionable civilian safety intelligence. The system's three core algorithmic contributions --- the spatio-temporal UAV route reconstruction, the sliding-window departure time optimizer, and the Poisson-based Shower Index --- demonstrate that meaningful security analytics are achievable from passively collected public data, without specialized sensors or proprietary feeds.

The application's fully client-side architecture, seven-language support, offline capability, and PWA installation make it accessible to the broadest possible audience during high-stress security events. The codebase also demonstrates engineering lessons of broader applicability: the hazards of defining React components inside render functions, the importance of explicit pixel sizing for canvas-based charting libraries, and the value of conservative fallback strategies in latency-sensitive API calls.

We hope this work serves both as a publicly useful tool and as a demonstration that academic BI projects can produce production-quality, societal-value applications.

\section{Window Score Implementation}
\label{app:window}

\begin{lstlisting}[style=tscode, language=JavaScript,
  caption=getWindowScore: scores a departure hour over the full travel window]
function getWindowScore(
  hourCounts:  number[],  // 24-element: alerts per hour of day
  startH:      number,    // departure hour 0-23
  durationH:   number     // trip duration in hours (may be fractional)
): number {
  if (durationH <= 0) return hourCounts[startH % 24];
  let score = 0;
  const fullHours      = Math.floor(durationH);
  const partialFraction = durationH - fullHours;
  for (let i = 0; i < fullHours; i++) {
    score += hourCounts[(startH + i) % 24];          // each full hour
  }
  if (partialFraction > 0.001) {                     // fractional last hour
    score += hourCounts[(startH + fullHours) % 24] * partialFraction;
  }
  return score;  // lower = safer departure window
}
\end{lstlisting}
